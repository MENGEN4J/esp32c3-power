# 变更工作区

> 对于复杂功能（跨多文件、涉及架构变更），先在此目录产出规划文档，再实现代码。

## 目录结构

```
changes/
└── <变更名称>/
    ├── proposal.md    ← 为什么要改、改什么
    ├── design.md      ← 技术方案
    └── tasks.md       ← 任务拆解
```

## 工作流程

1. **Proposal**：描述变更动机和范围，回答"为什么要改"
2. **Design**：描述技术方案，回答"怎么改"
3. **Tasks**：拆解为可独立实现+测试的原子任务
4. **实现**：逐个完成 tasks.md 中的任务
5. **Archive**：完成后将关键决策归档到 `specs/`，删除 `changes/` 下的工作文件

## 何时使用

- 跨 3 个以上文件的修改
- 涉及架构变更（如新增模块、修改模块职责边界）
- 涉及数据格式变更（如配置文件格式迁移）
- 涉及硬件交互变更（如 BLE/WiFi 互斥逻辑调整）

## 何时不使用

- 单文件 bug 修复
- 配置常量调整
- 日志/注释修改
- HTML 页面样式调整
