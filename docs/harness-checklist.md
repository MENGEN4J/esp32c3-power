## 一、上下文管理（对应 Claude Code 上下文架构层）

> 核心原则：智能不是瓶颈，上下文才是。不是越长越好，而是越精准越有力。

### 1.1 iteration_state.md 精简度

| 检查项 | 当前状态 | 差距 | 改造方向 | 优先级 |
|--------|---------|------|---------|--------|
| 行数是否 ≤ 120 行 | ✅ 42 行 | 无 | 保持 | - |
| 每行是否"去掉就会犯错" | ⚠️ 部分行偏描述性 | 需精简 | 删除非约束性描述，只保留硬规则 | P1 |
| 是否区分 IMPORTANT / YOU MUST | ✅ 已区分 | 无 | 保持 | - |
| 是否与 MEMORY.md 内容重复 | ⚠️ Harness 七层速查表与 MEMORY.md 有重叠 | 需去重 | 速查表保留在 iteration_state.md，详细说明移至 MEMORY.md | P1 |

### 1.2 规则文件分层与 Prompt Cache 优化

| 检查项 | 当前状态 | 差距 | 改造方向 | 优先级 |
|--------|---------|------|---------|--------|
| 固定规则与动态状态是否分离 | ⚠️ code_style.md 中混有运行时限制和命名规范 | 未按"固定/动态"分层 | 将不变的编码规范（命名、格式）与可能变化的运行时限制（内存数值、模块列表）分离 | P2 |
| 规则文件总 token 是否可控 | ⚠️ 9 个规则文件，总量较大 | 未评估 token 开销 | 测量各文件 token 数，总量超 8000 token 时考虑合并或精简 | P1 |
| 是否存在"上下文腐化"风险 | ⚠️ 规则文件间有重复内容 | 重复=浪费 token | 消除跨文件重复，每个规则只出现一次 | P1 |

### 1.3 上下文压缩策略

| 检查项 | 当前状态 | 差距 | 改造方向 | 优先级 |
|--------|---------|------|---------|--------|
| 是否有 compact 前的关键信息保护 | ❌ 无 PreCompact hook | Claude Code 有 PreCompact hook 减少 30% 关键信息丢失 | 在 auto_hooks.md 中定义 compact 前必须保留的信息列表 | P1 |
| 是否有压缩失败熔断器 | ❌ 无 | Claude Code 源码：连续失败 3 次就停 | 定义 compact 失败后的降级策略（如手动 review） | P2 |
| 是否有结构化 prompt 规范 | ❌ 无 | 结构化 prompt 比叙述式少消耗 30% token | 将规则文件中的叙述式描述改为结构化表格/列表 | P1 |
