标准发版模板

适用于本仓库后续所有版本发布。原则是：commit 写简短标题，详细内容写到 CHANGELOG 和 Gitee Release 说明。

---

一、使用方式

每次发版按下面顺序填写：

  1. 先更新版本号、日期、分支和附件清单
  2. 填写 CHANGELOG.md 中当前版本条目
  3. 填写 Gitee Release 标题和说明，必须包含固定章节，不能只写简短摘要
  4. 上传发版附件，至少包含 target/firmware.bin
  5. 补充测试结果、已知限制和未验证项

---

二、发版前检查清单

复制下面清单，逐项确认：

## 发版前检查

- [ ] 当前代码已合并到 `main`
- [ ] 版本号已按 SemVer 更新：`vMAJOR.MINOR.PATCH`
- [ ] `commit message`、`git tag`、Gitee Release 版本号一致
- [ ] 已更新 `CHANGELOG.md`
- [ ] 已准备 Gitee Release 说明，包含版本概览、本版重点、详细变更、测试报告、下载文件说明、使用说明、兼容性说明、已知限制、回滚说明、附件清单
- [ ] 已确认附件包含 `target/firmware.bin`
- [ ] 已确认附件包含 `v<版本号>-ota.zip` 和 `releases/latest/version.json`
- [ ] 已生成或更新 `releases/latest/version.json`
- [ ] 已确认 `.mpy` 文件目录与版本号一致
- [ ] 已完成语法检查
- [ ] 已完成固件编译或打包验证
- [ ] 已完成部署或真机验证
- [ ] 已补充未验证项和已知限制

五、Gitee Release 模板

下面模板面向下载用户，内容要比 CHANGELOG 更偏“我该下载什么、怎么用、有什么风险”。

强制要求：

  • Release 正文不能只写 1-3 行概述。
  • Release 正文必须包含下面模板中的所有一级章节。
  • 如果 Release 已存在，补发附件前必须同步更新 Release 正文。
  • 下载文件说明必须写清每个附件的用途、适用场景和推荐使用方式。
  • 测试结论必须和 CHANGELOG.md 中的新版本测试报告一致。

六、推荐补充信息

如果版本改动比较大，建议额外补这些字段：

  • 适合谁升级：普通用户、开发者、旧版本用户、遇到某个特定 Bug 的用户
  • 不建议谁升级：依赖未验证功能的用户，或当前环境不满足条件的用户
  • 与上一版本对比：这次是修复版还是行为变更版
  • 升级耗时：大致需要几分钟，是否会自动重启
  • 截图或录屏：适合网页交互、OTA 流程、配置页面变更
  • 哈希和大小：适合正式发版时做校验信息
