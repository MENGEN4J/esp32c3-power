一、核心概念

1.1 Harness（马具）

LLM 是一匹动力十足的烈马，Harness 就是让它全速奔跑、又不偏离轨道的缰绳。

个人 Agent = 通用 Agent + 个性化上下文 & 流程约束

组件 做什么
Context Engineering 告诉 AI 工作环境、技能、工作方式
TDD + 静态分析 每次修改都能被机械化验证
架构约束 什么能做、什么不能做
执行隔离 Sandbox，物理保障
Entropy 治理 定期扫描文档 drift、命名漂移、死代码堆积

1.2 Vibe Coding vs Spec Coding

维度 Vibe Coding Spec Coding
工作方式 Prompt → AI 猜测 → 反复修改 先写规格 → AI 按规格生成
AI 角色 决策者 + 执行者 纯执行者
人类角色 反复纠错的 debugger 架构师 + Spec 设计者
可控性 低 高——规格即契约
可维护性 差 好——Spec 本身就是活文档

1.3 三层上下文控制

层级 文件 作用
全局层 project.md 技术栈版本、架构模式、编码规范——AI 的"世界观"
指令层 AGENTS.md 行为准则、读取顺序——强制 AI 进入 Workflow
任务层 specs/ + changes/ 当前系统增量变更 + 本次变更的精确范围——按需读取

1.4 三层规范体系

层次 作用 本项目对应
约束层 告诉 AI「禁止什么、必须怎样」 .trae/rules/ 目录
示范层 告诉 AI「标准产出长什么样」 ✅ .trae/rules/templates/ 目录
视觉层 告诉 AI「页面应该长什么样」 web/ 目录（部分）

四、关键洞察

  1. 个性化上下文是复利工程：你用 Harness 越久，上下文越丰富，Agent 表现越好。这不是模型变聪明了，而是 Harness 变厚了。
  1. Spec Coding 的本质是增量知识库：每做一个需求自动积累一份文档，而不是另外花时间写。
  1. 约束层只是底线，示范层才是上限：只有「禁止什么」时，AI 知道不能做什么但缺乏参考；加入「标准产出长什么样」后，输出质量和一致性明显提升。
  1. 通用 Harness 是临时解法：随着模型变强，通用约束会被拆掉。但个性化上下文不会——它只会越来越厚。
  1. 复杂问题的结构性边界：云环境构建失败、多根因互相掩盖、隐性行为无文档——这类问题超出了 AI 的信息范围，需要人类介入。
